# Project Rules

## Architecture

- Follow SOLID principles for all components, services, hooks, and modules.
- Prefer composition over inheritance.
- Keep components small, reusable, and single-responsibility.
- Separate business logic from UI logic.
- Avoid massive files and deeply coupled modules.

---

# Naming Conventions

## Use kebab-case for:

- folders
- file names
- route names

Examples:

- user-profile-card.tsx
- training-module-service.ts
- payment-history-table.tsx

---

# Component Rules

- One component per file.
- Shared reusable UI goes into shared/components.
- Feature-specific UI stays inside the feature folder.
- Components must remain presentation-focused whenever possible.
- Move business logic into hooks/services.

---

# Folder Structure

Use feature-first architecture.

All files MUST live inside their proper feature module.

## Required feature structure

src/features/<feature-name>/

- components/
- hooks/
- services/
- store/
- lib/
- pages/
- types/
- utils/
- utils/constants/
- validators/

Each folder should contain only files related to that feature.

---

# Feature Generation

All new features should follow the generated structure from the project feature generator script.

Do not create ad-hoc folders outside the approved structure.

---

# Hooks

- Hooks must start with `use`.
- Hooks should encapsulate reusable logic.
- Avoid mixing UI rendering with hook logic.

Examples:

- use-auth.ts
- use-training-module.ts

---

# Services

- Services handle:
  - API calls
  - database interaction
  - external integrations
  - business logic

- Keep services framework-independent whenever possible.

---

# Types

- Shared feature types go into `types/`.
- Avoid inline large TypeScript types inside components.

---

# Validation

- Use Zod for validation.
- Validators belong inside `validators/`.

---

# State Management

- Feature-local state belongs inside feature stores/hooks.
- Avoid unnecessary global state.

---

# Import Rules

- Prefer absolute imports.
- Avoid circular dependencies.
- Shared modules belong in shared/.

---

# Clean Code Rules

- Avoid duplicated logic.
- Prefer explicit typing.
- Keep functions focused and small.
- Use descriptive naming.
- Remove dead code immediately.

---

# UI/UX Rules

- Prioritize accessibility.
- Maintain responsive layouts.
- Use consistent spacing and typography.
- Prefer reusable UI primitives.

---

# Performance Rules

- Lazy load heavy modules when appropriate.
- Avoid unnecessary re-renders.
- Memoize expensive computations carefully.

---

# Backend/API Rules

- Keep controllers thin.
- Business logic belongs in services.
- Validate all external input.
- Use proper error handling.
- Keep database access centralized.

---

# Prisma Rules

- Use relations properly.
- Prefer normalized structures.
- Avoid duplicated metadata fields.
- Use indexes intentionally.
- Keep models scalable and extensible.

---

# File Upload Rules

- Store metadata in File model.
- Store public URLs separately where needed.
- Keep storage provider logic abstracted.

---

# General Engineering Standards

- Code should be scalable.
- Code should be maintainable.
- Code should be production-grade.
- Optimize for long-term readability over shortcuts.
